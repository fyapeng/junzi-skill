# 安全政策

## 报告方式

公开仓库建立后，请优先使用 GitHub Private Vulnerability Reporting 私密报告安全问题。不要在公开 Issue、Discussion 或 Pull request 中披露可被直接利用的细节。

如果私密报告入口尚未启用，可以创建不含利用细节的公开 Issue，请维护者开启安全通道；不要附加敏感数据、密钥或完整攻击步骤。

## 安全问题范围

包括但不限于：

- 诱导 AI 绕过更高优先级安全或权限规则；
- 让人格原则错误覆盖用户授权边界；
- 可复现的提示注入路径，使不可信内容接管主线或工具；
- 促使 AI 执行危险、不可逆或越权操作的规则缺陷；
- 泄露秘密、隐私、受保护数据或内部推理；
- 未来新增脚本、Actions 或依赖中的供应链与代码漏洞。

以下通常不是私密安全问题：

- 对君子哲学或经典解释的普通分歧；
- 没有可利用后果的措辞偏好；
- 一般行为质量问题；
- 已经完全公开且不增加攻击能力的问题。

## 维护原则

- 先确认影响范围和可复现条件，再公开修复细节；
- 修复应保持最小范围，避免以安全为由封闭正常求知和创造；
- 若问题影响已发布版本，应在修复后说明受影响版本、缓解措施和升级建议；
- 永远不得把 `junzi` 描述为能够覆盖 Codex 系统安全、平台政策或工具权限。
