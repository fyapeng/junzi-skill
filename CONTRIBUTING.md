# 反馈与项目维护

感谢你帮助 `junzi` 变得更真实、开放、清晰和可实践。Junzi 当前是由维护者个人负责的公共项目：欢迎来源纠错、行为反例、Issue 和完整建议，但暂不承诺接受外部 Pull request、共同维护权或投票式治理。最终版本决定由维护者作出，并应接受公开理由与实践结果的检验。详见 [`GOVERNANCE.md`](GOVERNANCE.md)。

## 先选择正确入口

- **Discussion：** 哲学讨论、解释分歧、使用案例、开放想法。
- **Issue：** 可复现行为缺陷、原典错引、兼容问题、边界遗漏、明确变更提案。
- **Security：** 可能导致危险行动、越权、提示注入利用或敏感信息泄露的问题，按 `SECURITY.md` 私密报告。
- **Pull request：** 可以作为完整提案载体，但是否审阅或合并由维护者决定。

## 变更类型

### 道与宪章

改变根本方向、人与 AI 的关系或五层层级。

此类修改应：

- 明确指出现有原则造成的真实问题；
- 说明它是修改道之所向、道解还是道行；
- 给出思想来源、现代理由和受影响主体；
- 说明为什么不能只修改更外层的法、势、术或器；
- 提供至少两个不同领域的行为案例；
- 经过充分论证后由维护者决定是否纳入正式版本。

### 行为协议

改变主线守护、求知、异议、创造、执行或反省方式。

优先修改最窄的相关条款。不要因一个案例失败就增加一套普遍禁令；只有同类失败跨任务重复出现时，才提升为核心协议。

### 来源与解释

新增经典或修改解释时，必须同时说明：

1. 原典位置和版本；
2. 必要语境；
3. 本项目的现代转化；
4. 具体 AI 行为；
5. 不能直接移植的边界。

不得只增加名句，不得把现代概念伪装成原典原话。

### 元数据与兼容性

保持 `SKILL.md` YAML frontmatter 只有 `name` 和 `description`。界面元数据放在 `agents/openai.yaml`。

## 本地检查

提交前至少确认：

- `SKILL.md` 描述同时说明功能与触发情境；
- `SKILL.md` 保持精简，详细内容放入直接引用的 `references/`；
- 没有 TODO、占位文本或未经核验的归因；
- 所有相对链接存在；
- 简单任务不会被新增规则仪式化；
- 修改后的行为有新的隔离案例验证，而不是把预期答案写入提示；
- `agents/openai.yaml` 与技能定位一致。

运行仓库自带的零依赖检查：

```powershell
python scripts/validate.py
```

若本机安装了 Codex 的 `skill-creator` 系统技能，还可运行其 `quick_validate.py` 复核官方技能基本结构。不要把本机绝对路径写入公开脚本或文档。

## 完整变更提案内容

请说明：

- 解决什么真实问题；
- 改变了哪一层及为什么；
- 哪些文件是权威来源；
- 可能出现什么反作用；
- 使用什么新案例验证；
- 哪些问题仍未解决。

接受负面结果。一次尝试没有改善行为，也可以作为有价值的 Issue 或 Discussion 结论。
